# 全国スケーリング計画 (草案)

ステータス: 草案 (2026-07-16)
目標: 全国 1,741 市区町村の半数 (約 870) のごみ収集カレンダーを、現実的な期間で収録する。

方針 (2026-07-16 確定): **本データは純粋なオープンデータ (CC BY 4.0) として誰でも使える状態を維持し、
データ自体の収益化 (API 課金・SLA 販売等) は行わない。** 収益化はダウンストリームのアプリ側
(毎日使うごみアプリを入口に、店・コミュニティと利用者をつなぐ層) で考える。

## 0. 前提となる知見 (9 自治体の実績から)

1. **コストの大半は「都市固有」ではなく「ソース形式固有」** — CSV/HTML/PDF の 3 パイプライン
   (csv-extractor / html-extractor / pdf-extractor) が既にあり、同型ソースの 2 都市目からは
   設定と軽い差分コードで済む。中野 (CSV) は照合ツール込みで実質 1 日、同型なら数時間。
2. **検証は独立ソースの相互照合で機械化できる** — 中野は CSV × 現行 HTML × コース別 PDF の
   三重照合を全自動化し、全 42 町丁目 × 通年一致まで持ち込めた。一部を開封検証すれば残りの
   正しさが統計的に保証される (cut-and-choose)。全数照合が無理な自治体でもサンプル照合で足りる。
3. **配布物側の誤りは実在する** (中野区 PDF の日付ズレ印字等)。errata として記録する運用が要る。
   逆に言えば、独立ソースが 1 つしか無い自治体は「未検証」ラベルで出すしかない (鯖江方式)。
4. **例外日 (年末年始・特別収集) が構造的な保守課題** — 通常ルールは安定 (中野は 5 年不変) だが、
   年末年始は毎年 12 月頃に各自治体が個別発表する。収録数に比例して毎年 12 月に更新作業が発生する。
5. **探索の起点は整備済み** — デジタル庁の都道府県カタログ一覧 + 県ポータル検索
   (docs/opendata-sources.md)。県が全市町村分を共同公開する例 (福井 17 市町) は一括収録できる。

追記 (2026-07-17、杉並・川崎・調布の並行収録から):

6. **自治体単位のエージェント並行収録が機能する** — 1 自治体 1 エージェント (調査→抽出→照合) +
   統括者のレビュー・統合で、3 自治体 (計 112 コース、うち政令市 1) を 1 日で収録・照合まで完了。
   作業が `tools/*/<handle>/` と `municipalities/<pref>/<handle>/` に閉じるため衝突せず、
   Wave 量産の実行モデルとしてそのまま使える。共有部分 (語彙・schema) の変更だけ統括者が一元処理。
7. **ソース型の序列が見えた** — 最良は「テキスト版カレンダー」(調布。日付入り通年データを直接照合、
   triage で最優先確認)。次点が OD/サイト内 CSV (中野・杉並)、HTML 表 (練馬・川崎)。
   最悪はテキスト層なし PDF (川崎のカバー PDF = Print To PDF 製。pdftotext 不可、
   照合はサンプリング打ち切り規則で対処)。検証の信頼度と打ち切りの基準は
   docs/opendata-sources.md「検証の考え方 (確率論的な信頼度)」に明文化済み。
8. **政令市の区表現は確定済み** (川崎: course slug = `<区romaji>-<n>`、区またぎ同名町は「町名（区名）」)。
   残る 19 政令市 (人口 KPI の中核) はこの先例に従う。

## 1. ゴール定義と優先順位

- **KPI は「自治体数」と「人口カバー率」の二本立て**にする。人口上位 ~130 市区で全国人口の
  約半分に届くため、体感インパクトは自治体数より人口カバー率が支配する。
- 優先順序: (1) 人口降順 × (2) ソース難易度昇順 のスコアで貪欲に。
  県共同公開・標準スキーマ採用自治体はスコア無視で先に一括収録 (限界コストが最小)。
- 対象外を明示する: 紙配布のみ・Web に日程情報なし・毎年地区割が大きく変わる等の自治体は
  「収録不能」として台帳に理由付きで記録し、半数達成の母数から管理する。

### 台帳 (Phase 1 の成果物)

`docs/triage/<都道府県>.yaml` に全自治体分:

```yaml
- code: "13114"
  handle: tokyo-nakano
  population: 344880
  source_type: csv        # csv | html | pdf | image-pdf | app-only | none
  schedule_url: ...
  verify_sources: [html, course-pdf]   # 独立検証源の有無が品質ラベルを決める
  license: CC-BY-4.0
  difficulty: 2           # 1=県共同/標準OD 2=素直なCSV/HTML 3=機械可読PDF 4=画像PDF 5=収録不能
  notes: 年末年始は12月告知
```

台帳自体を公開データにする (「全国ごみ収集日オープンデータ カバレッジマップ」として単体でも価値がある)。

### 共通基盤 (Phase 2)

- 現状 3 パイプラインの共通部を `tools/_lib/` へ抽出: 曜日/第n週パーサ、シグネチャ畳み込み、
  年末年始 overrides 生成、yomi 解決 (郵便番号データ由来の町名読み辞書で自動化)、
  schema 検証、照合ハーネス (rules 展開 × 独立ソースの diff)。
- 都市追加を「`sources.yaml` を書く + 形式固有の差分コード最小」に落とす。
- CI: 全自治体の fetch 再実行 → 出力 diff 検知 (既存 golden 方式の一般化)。
  ソース URL 死活監視 (墨田区のようなリンク切れは自動検知)。

## 3. フェーズ計画と見積もり

### Phase 1: 全国 triage (1〜2 ヶ月)

- 47 都道府県 × カタログ/公式サイト走査を LLM エージェントで並列化 (1 県 = 1 バッチ)。
  1 自治体あたり数分の機械調査 + 県単位の人間レビュー。
- 完了時に難易度分布の実数が出る (以降の見積もりを更新する一次情報)。
- 副産物: 県共同公開・標準スキーマ採用の即収録リスト。

### Phase 2: 共通基盤化 (Phase 1 と並行、実働 2〜4 週)

- 既存 6 自治体を共通基盤上に載せ替えてリグレッションゼロを確認 (lift & shift → 分割)。

### Phase 3: 量産 (主工程、12〜18 ヶ月を想定)

Wave 制で同型をまとめて処理する:

- **Wave A — 県共同・標準スキーマ OD** (推定 100〜200 自治体): パーサ 1 本/県 or 1 本/標準。
  1 バッチで数十自治体。数週間で完了しうる。
- **Wave B — 素直な CSV/HTML** (推定 300〜400): 中野・練馬型。1 自治体 2h〜1 日。
  同型 10〜20 自治体を 1 バッチとして LLM エージェント並列 + 機械検証 green + サンプル目視。
- **Wave C — 機械可読 PDF** (推定 400〜600): 飯能・日高型。1 自治体 1〜3 日と最も重い。
  人口上位のみ着手し、半数達成に必要な分だけ下ろしていく。
- **Wave D — 画像 PDF・アプリのみ** : 原則後回し。OCR/目視コストが機械検証を上回る間は見送り。

スループット試算 (Phase 1 で実数に更新):
楽観 = A:150 + B:400 + C:320 で 870 到達、実働 ≈ 500〜900 人日相当。
エージェント並列化で暦時間はレビュー律速になり、**12〜18 ヶ月**が現実線。
人口カバー 50% (上位 ~130) だけなら **3〜4 ヶ月**で先行達成できる。

### Phase 4: 保守 (常設)

- **年度切替 (2〜3 月)**: 全自治体の fetch→build→verify を CI で一斉再実行。
  変化なし自治体は無人で年度更新、diff が出た自治体だけ人間レビュー。
- **年末年始 (11〜12 月)**: 各自治体の告知ページ (meta.yaml yearend_url) を巡回して
  overrides を確定させる年次キャンペーン。ここだけは構造的に人手+LLM の混成が要る。
- 保守コストは収録数に比例する。**量産と同時に保守自動化率を KPI にする**
  (「無人で年度更新できた自治体の割合」)。これが低いままだと半数収録は維持できない。

## 4. 品質ポリシー

- 3 段階ラベル: **verified** (独立ソース照合済) / **cross-checked** (2 ソース一致) /
  **unverified** (単一ソース、鯖江方式で明示)。
- 検証はサンプル照合で足りる (cut-and-choose): 全コース照合が安ければ全数、
  重ければ無作為 + エッジ (第n週・土曜収集・年末年始境界) を開封。
- 配布物側の誤り (errata) は自治体別に記録・公開。将来は自治体への報告も検討。

## 5. リスク

- **難易度分布が想定より悪い** (PDF/画像偏重) → Phase 1 で早期に判明。半数目標の期限を
  人口カバー KPI に振り替えて調整。
- **保守の複利** — 収録が増えるほど 12 月・3 月の負荷が増える。自動化率が上がらないなら
  収録ペースを落とすガバナンスを入れる。
- **ソース側の仕様変更・サイトリニューアル** — drift 監視 CI で検知し、都度パイプライン修正。
- **ライセンス** — OD は CC BY 系で問題なし。公式 HTML/PDF 由来は事実データ (日程) の抽出で
  著作権対象外だが、出典明記と取得マナー (レート制限) を全パイプラインで徹底。

## 6. 直近のネクストアクション

1. Phase 1 のパイロット: 1 県 (例: 埼玉 63 市町村) を丸ごと triage して所要時間と分布を実測
2. `tools/_lib/` への共通部抽出 (中野 + 練馬の重複からボトムアップで)
3. 標準スキーマ (デジタル庁推奨データセット系・入間型「日付単位 CSV」) の採用自治体数を調査
4. 郵便番号データ由来の町名 yomi 自動解決の PoC (yomi.yaml 手書きの排除)

## 7. コスト試算 (2026-07-16 追記)

実測アンカー: 中野区 (CSV型・照合ツール新規開発込み) ≈ エージェント実働 1 日。
同型 2 都市目以降はツール再利用で大幅に下がる前提。金額は 1$=150 円、
LLM API はモデル混成 (機械的工程は安価モデル) を想定。

### 初期構築 (半数 870 自治体、分布想定 A:150 / B:400 / C:320)

- Phase 1 triage: 全 1,741 走査。エージェント 20〜40 日相当、API ¥15〜50 万、人間レビュー 50〜100h
- Phase 2 共通基盤: エージェント 10〜20 日相当、API ¥8〜30 万
- Phase 3 量産:
  - Wave A (県共同/標準OD 150): パーサ十数本の償却で ~20 エージェント日、API ¥15〜30 万
  - Wave B (CSV/HTML 400): 0.25〜0.5 日/都市 → 100〜200 エージェント日、API ¥30〜120 万 (¥750〜3,000/都市)、レビュー 100〜200h
  - Wave C (PDF 320): 1〜3 日/都市 → 320〜960 エージェント日、API ¥100〜400 万 (¥3,000〜12,000/都市)、レビュー 160〜320h

**初期構築 合計**:
- LLM API: **¥170〜630 万** (悲観側で Wave C が重いと +数百万)
- エージェント実働: 500〜1,250 日相当 (並列実行するので暦では 2〜4 ヶ月分)
- 人間レビュー: **300〜600 時間** — これが暦時間の律速。1 日 2〜3h 充当で 6〜12 ヶ月
- 暦: **12〜18 ヶ月** (プラン本文と整合)

外注換算なら人間分 ¥150〜600 万 (5〜10k円/h) が加わり、総額 **¥300 万〜1,200 万規模**。

### 参考: 人口カバー 50% 先行 (上位 ~130 市区のみ)

大都市は OD/HTML 率が高く Wave C が薄い。API **¥50〜150 万**、レビュー 60〜120h、暦 **3〜4 ヶ月**。

### 年次保守 (870 自治体到達後、毎年)

- 年度切替 (2〜3月): CI 無人再実行 + diff レビュー。API ¥15〜60 万、人間 30〜60h
- 年末年始 (11〜12月): 870 件の告知巡回。API ¥15〜60 万、人間 40〜80h を約 6 週間に圧縮 (最繁忙)
- ソース drift 対応 (通年): サイト改編等で年 5〜10% が要修理 → エージェント 20〜60 日相当

**年次合計: API ¥50〜150 万 + 人間 100〜200h/年**。データセットの「維持費」であり、
収録数に比例するため、保守自動化率が上がらない場合は収録拡大を止める判断基準になる。

### 感度が大きい変数 (Phase 1 パイロットで実測すべきもの)

1. Wave C (PDF) の比率と 1 都市あたり実働 — 総コストの 5〜7 割を支配
2. 人間レビューの単位時間 — サンプルレビューで済む率 (cut-and-choose の開封率)
3. モデル価格の低下 — 12〜18 ヶ月の間に API 費は下振れしうる

## 8. 埼玉パイロット実測 (2026-07-17)

Phase 1 の第一県として埼玉 63 自治体 (収録済み飯能・日高含む) を triage した。
台帳: [`docs/triage/saitama.yaml`](triage/saitama.yaml)。調査は 7 エージェント並行 (1 自治体 10 分見切り) で半日。

### 分布 (61 調査 + 2 収録済み)

- **difficulty**: 2 (素直な CSV/HTML/JSON) = 16 / 3 (機械可読 PDF) = 34 / 4 (画像PDF等) = 13 / **1 (県共同・標準OD) = 0** / 5 (収録不能) = 0
  (さいたま市は当初 d4 と暫定評価 → gomisuke の JSONP API を実取得して **d2 に是正**。2026-07-17 統括追試済み。
  逆に北本・鶴ヶ島は 2026-07-18 の granularity バックフィル実確認で **d4 に是正** — カレンダーPDF がテキスト層なし画像。
  鶴ヶ島の「テキスト版」は収集曜日変更有無の要約文のみで**調布型ではなかった**)
- **granularity** (2026-07-18 バックフィル): dates (日付入り通年) 40 / weekday-rules 20 / partial 1 / none 1 / unknown 1 —
  6 割超が日付入りソースを持つ (ほぼカレンダー PDF 形態)
- **source_type**: pdf 33、html 15、image-pdf 10、json 2 (さいたま・行田)、csv/xls/none 各 1
- **人口カバー**: 県人口 729 万のうち difficulty≤3 で **83%**。
  残りは d4 の上尾・春日部・戸田・富士見・加須・北本・鶴ヶ島など (いずれも画像 PDF)。
- さいたま市 (135万) の一次は第三者商用サービス「ごみすけ」の JSONP API
  (`admin.gomisuke.jp/app/0017/api/jsonp.php`、認証不要・全70エリア+15ヶ月カレンダー)。
  **商用ベンダ API 依存のため、収録時は利用約款・負荷配慮 (キャッシュ・低頻度取得) の確認が必要**。

### 見積りへの含意 (§7 の感度変数の実測値)

1. **Wave C (画像PDF等の重量級) は件数比 21% (13/61)** — 事前想定レンジの下限寄り。ただし
   さいたま市を筆頭に**人口比では 34%** を占め、人口 KPI を追うなら d4 の攻略 (アプリ API 解析・OCR) が避けられない。
2. **県共同データセット (difficulty 1) はゼロ** — 福井方式は埼玉に存在しない。全県でこの前提は置けない。
3. 好材料: 埼玉県 OD ポータルのライセンスは PDL 1.0 (CC BY 互換) で、**入間市は日付入りカレンダー CSV を県 OD で公開** (唯一の
   収集日程 OD)。行田市は CMS の JSON API、越谷市は音声対応版テキスト HTML、鶴ヶ島市はテキスト版カレンダー候補と、
   「機械可読の裏口」が 1 割程度の自治体で見つかる。triage でこれを拾えるかが収録コストを大きく左右する。
4. 一部事務組合が実施主体の町村が多数 (蓮田白岡衛生・秩父広域・小川地区衛生・児玉郡市広域・大里広域など)。
   **組合単位でまとめて収録すると限界コストが下がる** (福井方式の代替となる「組合方式」)。

### 実装時の罠 (収録着手前に必読)

- **三郷市は「第n週 = カレンダー週 (1日〜7日=第1週)」解釈**で、他都市標準の「第n回目の該当曜日」と異なる。
  同種の解釈差が他にもありうるので、収録時は必ず注記・カレンダー実日付で確認する。
- 新座市は可燃・資源が HTML で素直だが、不燃・有害の地区別日程は画像 PDF のみ (部分 d4)。
- 熊谷市は種別ごとにソースが分かれる (一覧表 PDF は 3 種のみ、缶ビン等は別ページ)。
- **羽生市は可燃・不燃の収集曜日が公式 Web に一切無い** (ガイドブックも空欄様式で「環境課へ電話照会」方式)。
  「Web に日程情報なし」の初の実例で、実質 d5 (収録不能) 級。台帳上は資源ごみ (日曜) のみ判明のため d4。
- 久喜市は令和8年12月から分別区分統一の移行期 — 収録タイミングに注意。

## 9. 東京都パイロット実測 (2026-07-18)

第二都県として東京都 62 自治体 (収録済み 中野・練馬・杉並・調布 含む) を triage した。
台帳: [`docs/triage/tokyo.yaml`](triage/tokyo.yaml)。7 エージェント並行 (粒度フィールド granularity を追加)。

### 分布

- **difficulty**: 2 = 20 / 3 = 31 / 4 = 10 / 5 = 1 (神津島村 = ステーション別曜日が非公開)。
  **d1 (県共同・標準スキーマ) は東京もゼロ** — 各区の OD CSV は単独区のバラバラなスキーマ (d2 扱い)
- **source_type**: pdf 37、html 9、csv 7、image-pdf 4、txt 2 (調布・西東京)、xls 1 (大田)、app-only 1 (豊島)、none 1
