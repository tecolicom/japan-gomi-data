# メタルール形式 設計草案 — 年度カレンダーではなく生成規則を正典にする

2026-07-20 起案。エディタ構想と iCal 等価性の検討から出た方向づけのメモ。実装前の叩き台。

## 動機

現行スキーマ (rules + overrides、年度単位) は「ある年度への射影」を収録している。
しかし自治体の運用の実体は年度をまたぐ生成規則であり、それを一度定義すれば
毎年のカレンダーは「休業期間」という年次入力だけで生成できるはずである。

実証: 飯能 A2 の金曜 (ペットボトル/飲料缶の隔週交互) は、現行データでは
monthly_nth [2,4] + 1 月の差し替え override 4 行で表現しているが、
「**非休業金曜日の交互サイクル**」という 1 つの規則として解釈すると、
1/1 休業によるスキップから 1 月の繰り下げ 4 件がすべて自動導出され、
2 月の第 2・4 金への再同期まで一致する。手書きしている例外は規則の射影だった。

## 既存標準に無いことの確認 (2026-07-20 時点)

- デジタル庁「自治体標準オープンデータセット」: ゴミ分別方法一覧はあるが収集日程の定義は無い
- ODP 縦持ち CSV (品川・鯖江系): 収集曜日+祝日運用+適用期間列まで。サイクル系規則・休業期間モデルは無い
- **RFC 5545 (iCalendar)**: 規則は暦に対してのみ定義され、「収集実施日の列に対する規則」を持たない。
  `FREQ=WEEKLY;INTERVAL=2` + `EXDATE` では除外日がサイクルを進めないため、
  飯能型の「休業でサイクルが繰り下がる」挙動は規則として表現不能 (列挙への退避のみ)。
  現行 YAML との関係も、規則部は RFC5545 語彙の部分集合だが、例外部は演算モデルが別
  (YAML=日単位・月次置換の暗黙規約 / iCal=イベント単位の EXDATE・RECURRENCE-ID)。
  詳細はセッション検討記録 (2026-07-20) を参照。

よって実用可能な既存形式は無く、新設する価値がある。

## 3 層モデル

1. **恒久メタルール** (品目×コース): 語彙は下記
2. **年次パラメータ**: 休業期間 (年末年始等) のみ。自治体が毎年 12 月に決定・告知する
   実務プロセスとちょうど一致する
3. **適用期間バージョン**: ルールに valid_from / valid_to。地区割再編・品目変更・曜日変更は
   新バージョンとして積む (品川 CSV の適用開始/終了列と同型)

## 語彙案 (恒久ルール)

- `weekly` — 毎週◯曜 (現行どおり)
- `monthly_nth` — その月 n 回目の該当曜日 (現行どおり。暦基準)
- `eligible_nth` — **非休業ベースの第 n ◯曜** (その月 n 回目の「収集実施」該当曜日)
- `cycle` — **非休業◯曜日に対する周期列** (例: ペット→缶→ペット→…)。
  anchor (起点日とその品目) が必須。休業日はサイクルを消費しない
- `monthly_specific` — 実日付列挙 (退避先。規則化不能なものが残ることは許容)

overrides は明示演算に再定義する:
- 全品目休止 (現行 cancelled 相当) / **品目別単日休止** (現行スキーマの欠落。cancelled+category を
  「その品目のみ休止」と解釈する後方互換拡張) / 品目差し替え・追加 (暗黙の月次置換規約は廃し、明示に)

## 推定 → 承認ワークフロー

1. 観測 (年度カレンダー) から複数のメタルール仮説を立てる
   (「第 2・4 金+例外」と「非休業金曜サイクル」は単年では区別不能なことがある)
2. **同一自治体の全コース**を独立観測として突き合わせる (飯能は 6 コース全部が繰り下げ一致=サイクル支持)。
   複数年度があれば休業期間の違いで仮説が分岐し、さらに確定する
3. 推定ルールで生成した年度カレンダーを公式発表と全日照合 (現行の検証と同じ)
4. **自治体へ照会し、認められたら authority: confirmed として正典化**。
   未承認のメタルールは生成・点検の道具に留め、推定ルールによる未来年度の公開はしない
   (「推測でデータを作らない」原則の適用)

## 互換性

- メタルール → 現行 v1 YAML (年度 rules+overrides) へのコンパイラを書き、
  下流 (build-ics、city.tecoli、検証) は当面無変更
- .ics 配信は現行どおり展開済み全日イベント (解釈をクライアントに委ねない判断は不変)
- エディタ構想 (override 編集+プレビュー) は v1 のまま先行してよい。
  メタルール対応はその次の段
