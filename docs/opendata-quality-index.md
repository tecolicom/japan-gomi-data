# ごみの日データ 使いやすさスコア

自治体が発出する「ごみ収集日」データが、外部からどれだけ活用しやすいかを 100 点満点で採点する指標。triage 台帳 (`docs/triage/<都道府県>.yaml`) から機械的に算出できるよう設計している。

目的:
1. 都道府県単位の実測比較 (第一弾: 埼玉県・東京都) と、その公表 (note 記事等)
2. 全国 triage の共通評価軸 (難易度 difficulty の主観を、分解された 5 軸に置き換える)

「オープンデータとして立派か」ではなく **「市民・開発者が実際にデータを使えるか」** を測る。
公式 OD カタログに載っていなくても、機械可読で取得できれば得点する (ライセンス軸で差が付く)。

## 5 軸 100 点

### A. 取得性 (Access) — 20 点: プログラムから直接取得できるか

| 点 | 基準 |
|---|---|
| 20 | 認証・JS 不要の直 URL (curl 一発で取れる) |
| 15 | 直 URL だが軽い障害 (カタログの CLI ブロック・文字コード等。実体には到達可) |
| 10 | 検索 UI/JS の裏のエンドポイントを特定すれば取得可 (さいたま市型) |
| 5 | アプリ・検索 UI のみでバルク取得が困難 |
| 0 | Web 上に無い (紙配布・電話照会のみ。羽生市型) |

### B. 機械可読性 (Format) — 25 点: データの形式

| 点 | 基準 (source_type 対応) |
|---|---|
| 25 | 構造化データ: csv / json / xls (API 含む) |
| 20 | 規則的テキスト: txt (テキスト版カレンダー)・音声対応版 HTML (調布・越谷型) |
| 15 | html (表) |
| 10 | pdf (テキスト層あり) |
| 3 | image-pdf (OCR 必要) |
| 0 | none / app-only でデータ実体に到達不能 |

### C. 粒度・完全性 (Granularity) — 20 点: どこまで書いてあるか

| 点 | 基準 (granularity 対応) |
|---|---|
| 20 | dates: 日付入り通年 (年末年始等の例外日込み) |
| 14 | weekday-rules + 例外日も機械可読 (yearend: calendar-explicit) |
| 10 | weekday-rules のみ (例外日は毎年の都度告知) |
| 5 | partial: 一部種別・一部地区のみ |
| 0 | none |

### D. ライセンス (License) — 20 点: 再利用の法的確実性

| 点 | 基準 |
|---|---|
| 20 | 明示的 OD ライセンス: CC BY / PDL 1.0 / CC0 等 |
| 10 | 通常サイト掲載・禁止明記なし (事実データの抽出は可能だが再配布保証なし) |
| 5 | 転載禁止の明記あり、または第三者商用インフラ経由のみ (権利関係が複雑) |
| 0 | データなし |

### E. 検証可能性 (Verifiability) — 15 点: 正しさを独立に確かめられるか

| 点 | 基準 |
|---|---|
| 15 | 独立した第 2 ソースで日付レベルの照合が可能 (地域別カレンダー等) |
| 10 | 独立した第 2 ソースで曜日レベルの照合が可能 |
| 5 | 単一ソースのみ |
| 0 | 鮮度不明・ソース間矛盾 (要調査のまま放置されている) |

## ランク

| ランク | 点 | 意味 |
|---|---|---|
| S | 90〜100 | 模範。OD ライセンス付き構造化データ+日付粒度 |
| A | 75〜89 | 活用容易。機械可読+十分な粒度 |
| B | 60〜74 | 活用可能。変換の手間はあるが確実に使える |
| C | 40〜59 | 要労力。PDF 解析など形式固有の実装が要る |
| D | 20〜39 | 困難。OCR・API 解析・目視転記クラス |
| E | 0〜19 | 事実上使えない (Web に無い等) |

## キャップ規則 (加点式の穴を塞ぐ)

軸の加点だけでは「肝心のデータが存在しない」自治体が周辺加点で上位に来てしまうため、欠落の質による上限を設ける:

- **granularity = partial** (一部の地区・期間・非基幹品目が欠落。例: 墨田の半期のみ) → **59 点 (C) を上限**
- **core_missing = true** (基幹品目 = 可燃ごみの日程が Web に無い) → **19 点 (E) を上限**。
  可燃はごみ出しの中心であり、これが取れないデータは周辺情報があっても実用にならない
- **granularity = none / source_type = none** (収集日程が Web に無い) → **19 点 (E) を上限**

例: 羽生市は資源ごみの曜日だけ Web にあり加点合計は 65 相当だが、可燃・不燃が非公開 (core_missing) のため E。

## 算出規則

- `scripts/triage-score.mjs` が台帳から機械算出する。台帳フィールドとの対応は上表のとおり。
- unknown のフィールドは**その軸の中間より下の保守値**で仮置きし、`estimated` フラグを立てる
  (公表時は「暫定」と明示するか、確定させてから使う)。
- 台帳の機械判定を上書きしたい場合は台帳エントリに `score_overrides: {A: 15, ...}` を書く
  (理由を notes に)。人手裁定が常に優先。
- 指数は**一次ソースとして最良のもの 1 つ**について付ける (複数ソースがある場合は最高得点の組合せ)。

## 設計上の注意 (公表時に明記する)

- 本指数は 2026 年 7 月時点の**外部からの調査**に基づく。自治体内部の事情 (委託契約・システム制約) は
  評価対象でない。誤りの指摘は歓迎し、台帳を随時更新する。
- 10 分見切りの triage 由来のため、unknown を含む自治体のスコアは暫定。
- 「E. 検証可能性」はデータ利用者視点の軸であり、低くても住民向け情報提供が不十分という意味ではない。
- 指数が低い自治体を非難する趣旨ではなく、**どこを直せば市民・開発者がデータを使えるようになるか**
  (例: PDF に加えて CSV を 1 枚置く、ライセンスを明記する) を示すためのもの。
